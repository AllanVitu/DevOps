01

Vue d'ensemble du projet

Rytiger RPG (Les Chroniques des Oubliés) est un jeu de rôle dark fantasy développé intégralement en Vanilla JavaScript sans aucun framework d'interface. Le projet utilise Vite 5 comme outil de build moderne et Electron 41 pour empaqueter une version desktop Windows native. L'esthétique s'inspire de Dark Souls et Elden Ring : palette sombre, typographie pixel art (Press Start 2P, VT323), coins nets et ombres dures façon Minecraft.

Le jeu est distribué sous trois formes : une application web déployée sur GitHub Pages (PWA installable offline), un exécutable portable Windows via Electron, et un installateur Windows via electron-builder. Une sauvegarde cloud optionnelle s'appuie sur l'API REST Supabase (sans SDK — fetch natif uniquement).

Caractéristiques principales
  • 37 héros uniques avec 4 classes (Guerrier, Mage, Arche, Assassin) et 6 niveaux de rareté
  • 200 niveaux de campagne sur 10 zones organisées en 3 tiers de difficulté
  • Combat ATB (Active Time Battle) 4v4 — jauges de progression basées sur la vitesse SPD
  • Système de Break élémentaire : faiblesses, stagger, +50 % vulnérabilité
  • Dual gacha : portail héros + portail équipements avec garantie Épique au ×10
  • Forge de héros : fusion 2 héros → demi-dieu avec stats combinées
  • 480+ équipements sur 4 emplacements (Arme, Armure, Accessoire, Amulette)
  • 32 succès, missions journalières/hebdomadaires, connexion quotidienne avec streak
  • 5 modes additionnels : Donjon (5 étages), Horde, Raid, Tour de combat, VS PvP IA
  • Persistance duale : IndexedDB primaire + localStorage fallback + cloud Supabase
  • PWA offline, Service Worker, Gamepad API, haptics mobile, i18n FR/EN

5.1 GameState.js — État global (~800 lignes)

Singleton centralisant l'intégralité des données persistantes du joueur. Source unique de vérité (single source of truth) pour la progression, l'équipe, l'inventaire et toutes les statistiques.

  inventory: [{ instanceId, itemId }],

Méthodes clés
MéthodeDescription
init()Charge depuis IndexedDB avec fallback localStorage automatique

Persistance duale — Chaque save() écrit simultanément dans IndexedDB (base "rytiger_db" v1, store "kv", ~250 Mo) et localStorage (~5 Mo fallback). Au chargement, init() lit en priorité depuis IDB et tombe en fallback si la lecture échoue. Aucune donnée n'est perdue même en navigation privée Safari où IDB est restreint.

5.2 CombatEngine.js — Combat tour par tour (~600 lignes)

Moteur 4v4 en tours. Classe pure sans accès DOM — communique avec l'UI via des callbacks (onUpdate, onLog). Entièrement testable dans Node.js.

// Structure d'une unité de combat
{
  id, name, type, rarity, alive: true,
  stats: { hp, maxHp, atk, def, spd, magAtk, magDef },
  skills: [skillId1, skillId2, skillId3, skillId4],
  buffs:   [{ stat, boost, turns }],   // ex: { stat: 'atk', boost: 0.3, turns: 2 }
  debuffs: [{ type, damage, turns }],  // ex: { type: 'poison', damage: 15, turns: 3 }
  x, y                                 // Position sur la grille de combat
}
Mécaniques de combat
Calcul des dégâts

Physique : max(1, ATK − DEF) × puissance

Magique : max(1, magAtk − magDef) × puissance

Critique : ×1.5 selon stat CRIT

Système de Break

Frappe de faiblesse élémentaire → jauge de rupture++

Jauge pleine → Stagger → +50 % vulnérabilité N tours

Effets de statut

Poison, Brûlure : dégâts fixes par tour

Garde : −50 % dégâts reçus (action défensive)

Forge Gauge

Jauge combo d'équipe commune

Effets spéciaux déclenchés au maximum

5.3 ATBCombatEngine.js — Active Time Battle (~1000 lignes)

Implémentation principale du combat. Les unités agissent selon une jauge ATB qui se remplit progressivement en fonction de leur vitesse (SPD), simulant un temps semi-réel.

// Chaque frame de mise à jour :
fillRate = unit.stats.spd / 100          // SPD 90 → remplit 0.9 par unité de temps
unit.atbGauge += fillRate * deltaTime

// Quand la jauge atteint 100 :
unit.atbGauge = 0
unit.canAct = true                      // L'unité est prête à agir

// Après l'action, temps de récupération selon la compétence :
unit.atbGauge = -skill.recovery              // ex: -30 pour une attaque lourde
TypeStat offensiveStat défensiveDescription
physicalATKDEFAttaque physique standard
magicalmagAtkmagDefAttaque magique élémentaire
heal——Restaure des HP à une cible alliée
buff——Augmente une stat alliée pour N tours
debuff——Réduit une stat ennemie pour N tours
specialVariableVariableMécaniques uniques (clones, domaines, absorptions)

Routeur SPA hash-based minimaliste, sans dépendances. Singleton partagé. Transitions fade-out 150ms / fade-in 250ms entre écrans.

router.on('/home', renderHomeScreen)          // Enregistrement de route

router.current                                // Route active courante
router.start()                                // Init (écoute hashchange + route initiale)

  • Démarrage aléatoire (après premier événement utilisateur — politique iOS autoplay)

  • Volume maître (0–1) et volume SFX indépendants, persistés en localStorage

8.4 CloudSave.js — Sauvegarde Supabase (~150 lignes)

API REST Supabase sans SDK. Auth via Magic Link (lien par email, sans mot de passe).

  • requestMagicLink(email) — Envoie un lien de connexion par email
  • getSaveSlots() — Liste les slots de sauvegarde cloud disponibles
  • uploadSave(gameState) — Sérialise et uploade l'état complet (POST REST)
  • downloadSave(slotId) — Télécharge et restaure une sauvegarde (GET REST)
  • Config : VITE_SUPABASE_URL et VITE_SUPABASE_ANON_KEY dans .env
8.5 AssetUtils.js & SpriteAPI.js (~350 lignes)
  • Cache session mémoire : préfixe sprite_cache_v5_local_ (sessionStorage)
  • Chargement lazy via IntersectionObserver — marge 200px de pré-chargement
  • Fallback par type de classe si le sprite individuel est absent
  • 37 sprites uniques + 4 fallbacks par classe + boss/sbires
  • PNG 32×32 base, mise à l'échelle CSS (image-rendering: pixelated)
  • injectSpriteLazy(imgEl, type, charId) — injection différée pour les listes
8.6 InputManager.js (~100 lignes)
  • Gamepad API W3C — Xbox/PlayStation, mapping standard
  • Bouton 0 = Confirmer, 1 = Annuler, 2 = Action X, 3 = Action Y
  • Boutons 12–15 = D-Pad (haut, bas, gauche, droite)
  • Clavier : Flèches, Entrée, Échap
  • Anti-spam : cooldown 200ms entre deux actions
  • Mapping de boutons personnalisable

Patterns architecturaux

PATTERN 01
Singleton global

Trois singletons partagent l'état global : gameState (données joueur), router (navigation SPA), audioManager (BGM/SFX). Instanciés une fois au démarrage, importés par référence — pas de prop-drilling, pas de Context API.

PATTERN 02
Séparation logique / interface

Les moteurs de combat (CombatEngine, ATBCombatEngine) ne contiennent aucun accès DOM. La communication avec les screens se fait uniquement via des callbacks passés à l'initialisation (onUpdate, onLog, onBattleEnd). Cela rend le moteur 100 % testable dans Node.js et remplaçable sans toucher à la logique.

PATTERN 03
Persistance duale & résilience

Chaque save() écrit simultanément dans IndexedDB (primaire, ~250 Mo) et localStorage (fallback, ~5 Mo). Au chargement, init() lit en priorité depuis IDB. Aucune donnée n'est perdue même en navigation privée Safari où IDB est restreint.

PATTERN 04
Rendu par remplacement complet

Chaque renderXxxScreen(container) remplace intégralement innerHTML du container, puis attache les event listeners. Pas de VDOM, pas de diff, pas de composants à états internes. Simple, prévisible, sans fuites de listeners grâce au remplacement total.

PATTERN 05
Lazy-loading d'assets

Sprites chargés à la demande via IntersectionObserver avec marge anticipatoire 200px. Cache sessionStorage évite les rechargements inter-écrans. Fallback automatique vers le sprite générique du type de classe si l'image individuelle est absente.

PATTERN 06
ES Modules natifs (ESM)

Projet entièrement en type: "module". Aucun CommonJS dans le code source. Vite gère uniquement le bundling pour la production — en développement, les imports ESM natifs du navigateur sont utilisés directement, éliminant toute étape de transpilation et accélérant le HMR.
